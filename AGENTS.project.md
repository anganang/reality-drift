# 项目协作规则

目标是让系统始终可运行、可定位、可回退。本文件只保留所有模块都适用的仓库级规则；模块特有的协议、硬件和 checkpoint 约束放到对应目录的 `AGENTS.md`。

## 工作方式

非平凡改动前先明确目标、成功标准和影响边界。用户给宏观目标时，先问清是解决宏观问题还是定位具体路径做最小修复。

先读现有调用链、状态 owner 和初始化路径，再判断代码落点。状态 owner、协议格式、事件流或状态边界不清楚时先停下来说明。

优先做最小可运行链路：启动→初始化→主循环→核心 I/O→必要日志先跑通，再加高级功能、持久化、低功耗或外部控制。

涉及持久化存储、外部状态或不可逆配置时，先定义 rollback boundary：当前可构建提交或 diff、哪些状态会被写入、是否需要清除已有数据、如何确认回到已知状态。

连续 3 次以上代码修改且构建通过但未实际验证时，停下来要求验证当前整体行为，不继续纯代码推理。跨会话恢复时，先重新获取系统当前状态，不直接沿 `DEBUG_STATE.md` 的假设继续。

## 模块边界

入口文件只做初始化和模块编排。新增源码按职责进对应目录。

同一份状态只能有一个 owner。跨模块通信走明确事件或公开接口；依赖保持单向，出现循环依赖先调边界。

拆分模块时保持外部 API 稳定。清理范围限于：当前改动直接导致编译警告或运行时会引用到的死代码。

## Path-scoped Rules

根 `AGENTS.md` 只放所有模块都适用的规则。模块特有的协议细节、状态权威、资源约束、日志要求、checkpoint 和 rollback 边界，放到对应目录的 `AGENTS.md`。

新增模块目录时，如果该模块有独立状态 owner、协议格式、外部资源、持久化格式、时序约束或验证方式，必须同时创建或更新该目录的 `AGENTS.md`。

更新模块职责、协议、状态 owner 或验证路径时，优先更新对应目录的 `AGENTS.md`；只有跨模块通用的规则才上移到根文件。

## 运行时纪律

协议和状态先定再写代码。状态机、消息格式、接口契约、线程安全策略不能边写边决定。

一次只改一条主线。不盲目启用新特性、外部依赖或配置项；改基础设施先确认影响范围。

新增行为路径前先设计日志和错误码。日志至少能区分：初始化未到达、初始化失败、事件未进入、状态被忽略、协议收发异常、外部依赖未响应。影响核心链路的错误不能静默吞掉。

## Bug 处理

出现 bug 后先确认最后执行点，区分没执行、卡死、崩溃、行为异常；再用日志或二分法缩小到具体阶段。定位后只做一个最小修改验证。

权威状态优先级按模块语义确认：底层框架回调、持久化读回、明确 owner 的状态机、驱动/运行时返回值和实际系统行为优先于 UI、缓存值、自定义 marker 或派生日志。

需要 checkpoint 时选最靠近改动目标的验证点。任一 checkpoint 出现矛盾证据，先更新 phase 和假设，再决定下一步。

## 构建与验证

代码或配置改动后必须构建通过。文档-only 改动可以不构建，但要说明原因。行为改动必须给出验证方式和判断标准。模块边界、关键文件或排查日志变化时同步文档。

## 停止条件

以下情况必须停下来说明风险和下一步最小验证方式：

- 会破坏已验证的稳定主链路
- 当前假设无法解释最新行为
- 需要同时改多个独立子系统，但缺日志或依据
- 构建错误连续两次不是同一类简单错误

其余通用停止判断（信息不足、方向错误、rollback）由全局规则覆盖，不再重复。

## 跨会话状态

调试未完成时，将以下内容写入项目根目录 `DEBUG_STATE.md`：当前 phase、活跃假设、已排除路径、最近 rollback boundary、下一步验证点。下次对话开头读取并确认是否仍成立。调试完成后删除该文件。

## 删除限制

同全局规则：一次一个明确路径，批量删除让用户执行。

## 知识积累

调试或开发中发现以下类型的项目特有知识时，在当前任务完成后提议写入文档，用户确认后再写：

- 反复出现的故障模式和对应的首选排查路径
- 本项目特有的正确实现模式（不是通用最佳实践）
- 曾经走不通的方案和原因

不写通用知识、不写显而易见的事实、不写未经验证的猜测。每条记录必须附带"在什么场景下验证过"。
