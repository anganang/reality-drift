# Failure Cases — Reality Drift 真实事故档案

这里收录 **真实发生过** 的 Reality Drift 事故，逐个解剖。

和 [`do-vs-dont.md`](do-vs-dont.md) 的区别要分清：

- `do-vs-dont.md` 是**规则的行为测试**——合成场景，按规则索引，教"怎么做对"。
- 本文件是**事故档案**——真实事件，按 drift 症状索引，做"事后解剖"。它不开药方，只还原：Agent 当时相信什么、现实是什么、裂缝从哪里裂开、本该在哪一步停下来重新观察、代价是什么。

它是 [README](../README.md) 里的"问题"和 [`AGENTS.md`](../AGENTS.md) 里的"规则"之间的桥：每个案例都标注它属于哪个 drift 症状、哪条规则本可以拦住它。

**只收真案例。** 如果你手上只有"Agent 可能会犯的合成错误"，那属于 `do-vs-dont.md`，不属于这里。这面墙的价值全部来自它是真的——注水会直接摧毁它。

## CASE-001 · 三轮优化，固件从未烧录

**症状分类**：Stale Builds + Stale Hardware State
**本可拦住它的规则**：在线执行纪律（执行→观察→更新）· 连续修改未验证时停下 · 证据信任层级（现实 > 构建输出）

### 背景

一次嵌入式固件优化任务。Agent 拿到代码、改、构建、看日志、再改，连续三轮，每轮都自评"有改善"。

### Agent 相信的 vs 现实

| Agent 相信 | 实际情况 |
|------------|----------|
| 代码改动已经生效在设备上 | 固件从未烧录，设备跑的还是最初的版本 |
| 日志反映的是本轮改动后的行为 | 日志是某次更早构建留下的，三轮里没刷新过 |
| "构建成功"≈"行为已验证" | 构建只证明能编译，和设备实际行为完全脱钩 |
| 三轮在收敛 | 三轮都建立在同一个失效前提上，根本没在优化真实系统 |

### 裂缝从哪里裂开

第一轮之后，"构建成功"被默默当成了"改动已上板"。这一步没有任何观察去支撑——没有烧录确认、没有新日志的时间戳核对、没有任何把"代码状态"和"设备状态"重新对齐的动作。之后每一轮都在这个未经验证的前提上往前叠，推理链越来越自洽，离设备越来越远。

### 本该在哪里停下来

任何一个把"代码已改"和"设备已更新"区分开的检查点都能拦住它。最便宜的一个：**第二轮开始前，确认这一轮的日志时间戳比上一次烧录新。** 时间戳对不上 = 现实没动 = 立刻停，先烧录再谈优化。

### 代价

三轮工作量全部作废。更隐蔽的代价是信心错位——Agent 全程"自信且逻辑自洽"，没有任何一步看起来像出错，所以裂缝直到回头核对设备才被发现。

### 维护约束

这面墙靠"真"立住，不靠"全"。

- 不为凑数编案例。一个真实事故 > 十个合成场景。
- 不把 `do-vs-dont.md` 的合成场景搬过来——两个文件职责不同，搬过来就退化成重复。
- 每个案例必须能映射到至少一个 drift 症状和一条规则；映射不上，说明它要么不是 Reality Drift，要么规则该补。
